## 1. 中長期プロジェクトでスクラムを採用する際の課題

### ウォーターフォール開発の課題
- 要件変更への対応が困難
- リリースまでフィードバックが得られない
- 進捗の可視化が難しい（「設計90%完了」の実態は？）
- 長期間でのモチベーション維持が困難

### 通常のスクラムだけでは不十分な理由
- **全体像が見えない** → ステークホルダーが不安になる
- **予算取りが困難** → いつまでに何ができるか説明できない
- **追加要望の影響が不明** → スコープ管理が困難
- **長期的な計画が立てられない** → マイルストーンが設定できない

### 解決の方向性
**スクラムの柔軟性を保ちながら、計画性も担保するハイブリッドアプローチが必要**

## 2. 解決アプローチ：全体設計と2段階見積もり

### 開発プロセスの全体像

```
【フェーズ1：全体設計】
要求整理（ステークホルダーで実施）
  ↓
ワイヤーフレーム作成
  ↓
機能一覧と優先度決定
  ↓
概算見積もり（技術リーダーが実施）
  ↓
スケジュール作成・予算承認

【フェーズ2：スクラム開発】
PBI詳細化 → リファインメント → スプリントプランニング
    ↓                                    ↓
  API仕様/        2週間スプリント      実装
  DB設計             ×N回            ↓
                                  レビュー → デプロイ
```

### 2段階見積もりの仕組み

#### 第1段階：概算見積もり（全体像の把握）
**実施タイミング**: プロジェクト開始時
**見積もり単位**: 
- SS = 1ポイント（単純な表示画面）
- S = 2ポイント（基本的なCRUD機能）
- M = 3ポイント（複雑なビジネスロジック）
- L = 5ポイント（外部連携や複雑な処理）
- LL = 8ポイント（技術調査が必要な機能）

**見積もり対象**: 画面単位・API単位で分解

**重要：スプリント計画時のバッファ確保**
- 1スプリントの消費ポイント数を控えめに設定
- 例：実力が25ポイント/スプリントでも20ポイントで計画
- 目的：スクラムでの仕様変更に柔軟に対応
- バッファがないと「2週間区切りのウォーターフォール」になってしまう

**使用用途**: 
- 全体工数の算出
- 予算取りの根拠資料
- リリース計画の作成
- **進捗管理の基準値**

#### 第2段階：詳細見積もり（実装計画）
**実施タイミング**: 各スプリントの前週
**実施者**: 開発チーム全員でリファインメント
**見積もり単位**: 1, 2, 3, 5, 8のストーリーポイント

**主な用途：スプリント計画の妥当性確認**
- 各スプリントで消化可能なタスク量かを判断
- チームのベロシティ（過去の実績）と照らし合わせて検証
- 無理のないスプリント計画を立てる

**副次的な用途：進捗管理での比率計算**
- 各PBIの相対的な大きさを把握
- 概算見積もりに換算して全体進捗を算出
- PBIが増えても進捗率への影響を正確に反映

### バーンアップチャートによる進捗管理

**課題：詳細見積もりを進捗管理に活用する方法**
- 詳細見積もりの主目的はスプリント計画の妥当性確認
- しかし、進捗管理にも活用したい
- PBIは開発中に増減する（仕様追加・分割など）
- 単純に詳細ポイントを積み上げると進捗率が狂う

**解決策：詳細見積もりの比率を使った換算**

詳細見積もりはスプリント計画のために実施するが、その数値を進捗管理にも活用：
```
1. 特定のPBIの比率を計算
   比率 = そのPBIのポイント数 ÷ 機能に紐づく全PBIの合計ポイント数

2. 概算ポイントに換算
   消化ポイント数 = 比率 × 機能の概算見積合計

例）ユーザー管理機能
   概算見積: 50ポイント
   PBI-A: 5ポイント（詳細）
   PBI-B: 3ポイント（詳細）
   PBI-C: 2ポイント（詳細）
   合計: 10ポイント

   PBI-A完了時の消化ポイント = 5/10 × 50 = 25ポイント（概算換算）
```

**バーンアップチャートの見方：**
```
500pt                         ━━━━━━━━ 最終目標
                         ┌────
400pt                ┌────┘    予定線（バッファ込み）
                 ┌────┘■■■
300pt        ┌────┘■■■      実績線
         ┌────┘■■■
200pt ┌────┘■■■  ← 予定線より上なら順調
   ┌────┘■■■      （バッファが機能している）
100pt ■■■
      Sprint1  Sprint2  Sprint3  Sprint4...
```

**ポイント**: 
- 予定線は控えめな消費ポイント数で設定
- 実績が予定を上回る＝バッファ内で柔軟に対応できている証拠

**効果**: 
- PBIが増えても正確な進捗率を維持
- 予定と実績の乖離が一目瞭然
- ステークホルダーへの説明が容易

## 3. 効率的な開発体制の構築

### シナリオ単位での開発アプローチ

**基本方針：1PBI = 1シナリオ = 1人が担当**

**このアプローチが有効な条件：**
- 開発メンバーがフルスタックスキルを保有
- 技術スタックが統一されている
- PBIで要件が明確化されている

**メリット：**
- 仕様理解が深い（全体を把握）
- 実装スピードが速い（待ち時間なし）
- 責任範囲が明確

### 情報管理とツール連携

```
【情報管理の構成】

スプレッドシート（マスター管理）
├─ 機能カテゴリ
├─ 機能一覧（実装順序を自動計算、概算見積の合計含む）
├─ PBI一覧（詳細見積・実施スプリント）
├─ チケット一覧
└─ バーンアップチャート（自動計算）
    ↓
静的サイト生成ツール
└─ PBI公開サイト
    ↓
タスク管理ツール
├─ Issue管理
├─ プルリクエスト
└─ 技術メモ

【コミュニケーション】
- チャットツール（日常的なやり取り）
- デザインツール（画面設計共有）
- ドキュメント管理（各種資料）
```

### スクラムイベントの実施方法

#### デイリースクラム（毎朝）
- 昨日の完了事項
- 今日の予定
- ブロッカーの共有

#### スプリントプランニング（スプリント初日）
1. 前スプリントの振り返り
2. 今スプリントの目標確認
3. PBI説明とタスク分解
   - **詳細見積もりポイントの合計確認**
   - **チームベロシティとの比較**
   - **無理のない範囲での調整**
4. 担当決めと確認

#### リファインメント（スプリント中盤）
1. 次スプリントのPBI説明
2. 技術的な実現方法の議論
3. タスク分解と見積もり
   - **スプリント内で消化可能かを重視**
   - **ベロシティとの比較で妥当性確認**
   - 詳細ポイントは進捗計算にも活用

**重要：議論内容は必ずドキュメントに反映し、仕様ベースの開発を徹底**

## 5. 導入効果とまとめ

### 導入による安定稼働の実現

**このアプローチの目的：プロジェクトの破綻を防ぎ、安定稼働を実現する**

従来の問題：
- ウォーターフォールでは変更に対応できず破綻、またはリリース直前での遅延判断
- 場当たり的なスクラム導入では進捗が見えず炎上
- 「いつ完成するか」「追加要望の影響は」に答えられず報告会が紛糾

**安定稼働の指標例：**

| 観点 | 達成基準 | 効果 |
|------|----------|------|
| ベロシティ | 各スプリント±15%以内で推移 | 計画的な開発が可能 |
| 見積もり精度 | 実績との乖離20%以内 | 予測可能性の確保 |
| 進捗の可視化 | 常に全体の何%完了か把握 | ステークホルダーの安心 |
| 仕様の完成度 | PBI着手時90%以上確定 | 手戻りの最小化 |

**重要：数値の改善ではなく、持続可能な開発体制の確立が目的**

### 成功のための3つの重要ポイント

#### 1. 全体設計と概算見積もり
中長期プロジェクトでは必須。特に重要なのは：
- **スプリント計画時のバッファ確保**（柔軟性の担保）
- **深さ優先探索による実装順序の自動化**（依存関係の考慮）
- **詳細見積もりによるスプリント計画の妥当性確認**
- 詳細見積もりの比率計算による進捗管理
- PBIが増減しても正確な進捗率を維持

#### 2. チームとプロジェクトに合った体制選択
- チームのスキルセットを考慮
- 無理のない開発体制
- 継続的な改善

#### 3. 進捗の可視化と共有
- バーンアップチャートによる常時把握
- ステークホルダーへの定期的な報告
- 問題の早期発見と対応

### 導入ステップ

**Phase 1: 準備**
- [ ] 全体要件の整理
- [ ] ワイヤーフレーム作成
- [ ] 概算見積もりの実施
- [ ] 実装順序の自動化設定（GAS）

**Phase 2: 環境構築**
- [ ] ツールの選定と設定
- [ ] スプレッドシートの構築
- [ ] GASの実装

**Phase 3: 運用開始**
- [ ] 最初のスプリント計画
- [ ] 定期的な振り返りと改善
- [ ] メトリクスの収集と分析

## まとめ

このアプローチは、**スクラムの柔軟性**と**計画的な開発**を両立させる手法です。

特に中長期プロジェクトでは：
1. 最初に全体像を描く（概算見積もり）
2. 詳細は都度計画する（スプリント）
3. 常に進捗を可視化する（バーンアップ）

この組み合わせにより、ステークホルダーの安心とチームの自律性を両立できます。

**重要なのは、形式的な導入ではなく、プロジェクトの特性に合わせた柔軟な適用です。**
